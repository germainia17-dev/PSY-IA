# Rapport sécurité — Companion / PSY-IA

**Date : 13 juin 2026** · Périmètre : code client (Expo), Edge Functions, DB/RLS, Stripe, secrets.

Verdict en une ligne : **le cœur backend est correctement verrouillé. Tes deux vraies failles ne sont pas techniques — c'est une clé live que tu viens de cramer toi-même, et un mensonge écrit dans ton UI qui t'expose juridiquement.** Le reste, c'est du durcissement.

## 🔴 CRITIQUE

### C1 — Clé Stripe `sk_live_` compromise (par toi, aujourd'hui)
Tu as collé ta clé secrète **live** (`sk_live_51TDTd…`) en clair dans une conversation. Une clé secrète live = contrôle total de ton compte Stripe (créer des remboursements, lire tes clients, déclencher des paiements). Elle est désormais hors de ton contrôle.
- **Impact :** quiconque a la chaîne peut piloter ton Stripe en production.
- **Action (maintenant, pas après le test) :** Stripe → Developers → API keys → **Roll** la Secret key. Repose la nouvelle dans `supabase/.env.secrets`, re-push (`supabase secrets set --env-file …`). Le `whsec_` du webhook a aussi transité → régénère-le aussi (Webhooks → ta destination → Roll signing secret).
- **Constat brutal :** le risque n°1 de ce projet n'est pas un attaquant, c'est ta manipulation des secrets. Un secret ne se tape jamais dans un chat, un ticket, un commit. Tu as un fichier gitignored pour ça — utilise-le exclusivement.

### C2 — Mensonge « chiffré de bout en bout » (faille juridique, pas crypto)
`src/app/index.tsx:222` (« chiffré de bout en bout 🔒 »), `index.tsx:282`, `settings.tsx:77`, `store/listing-fr.md:13,24,55`.
- **C'est faux.** Aucun E2E n'est implémenté. Les messages sont en clair dans AsyncStorage et transitent en clair (déchiffrables côté serveur) vers Gemini/Anthropic. Ta propre `privacy-policy.md:17` dit la vérité (« relayé de manière chiffrée HTTPS/TLS ») — donc **ton UI contredit ta propre politique de confidentialité.** C'est l'aveu écrit.
- **Impact :** pratique commerciale trompeuse (DGCCRF, jusqu'à 10 % du CA / 300 k€), rejet App Store/Play, et c'est ton différenciateur central qui se retourne en bombe. Tu vends « privé » sur un mensonge vérifiable en 30 secondes.
- **Action :** remplace partout par une promesse vraie : **« Sans compte · Privé · Aucun tracker »** ou « Chiffré en transit (HTTPS) ». Ne réécris pas « E2E » tant que tweetnacl n'est pas réellement en place.

## 🟠 ÉLEVÉ

### H1 — Création illimitée de comptes anonymes = DoS sur ton portefeuille
`signInAnonymously()` (`src/lib/api.ts:27`) sans rate-limit ni CAPTCHA. Ton quota de 10 msg/jour est **par utilisateur**, et créer un utilisateur est gratuit et illimité.
- **Impact :** un script mint 10 000 sessions anonymes → 100 000 messages/jour sur ta facture Gemini/Anthropic. Le quota atomique en DB est bien fait, mais il protège le mauvais étage : il limite par identité, pas la création d'identités.
- **Action :** Supabase → Auth → active **CAPTCHA (hCaptcha/Turnstile)** sur les sign-ins anonymes + rate-limit par IP. Avant tout post public, c'est obligatoire.

### H2 — CORS `Access-Control-Allow-Origin: '*'`
`supabase/functions/chat/index.ts:8`. N'importe quelle origine web peut appeler ta fonction. Couplé à H1, n'importe quel site peut brûler ton quota depuis le navigateur de ses visiteurs.
- **Action :** restreins à tes origines réelles (ton domaine Vercel + `localhost` en dev) une fois l'URL web connue.

## 🟡 MOYEN

### M1 — Clé Anthropic potentiellement brûlée dans d'anciens builds
L'historique git actuel est propre (aucun `sk-ant`, aucun `EXPO_PUBLIC_ANTHROPIC_API_KEY` tracké). **Mais** si un APK/bundle a été distribué avec la clé en variable `EXPO_PUBLIC_*`, elle est extractible du binaire — git propre ne prouve rien sur les builds déjà sortis.
- **Action :** régénère-la sur console.anthropic.com par précaution (coût nul, bénéfice certain).

### M2 — Code promo 100 % non restreint en live
Un code 100 % en mode réel sans limite = Pro gratuit à vie pour quiconque le trouve/devine.
- **Action :** Stripe → limite à **1 utilisation** + expiration courte, ou supprime-le après ton test.

## ✅ Ce qui est BIEN fait (ne le casse pas)
- **Webhook Stripe :** signature HMAC vérifiée sur le corps brut, `client_reference_id` validé en UUID, `payment_status` exigé, idempotent, 500→retry. Solide.
- **RLS :** policies **SELECT only**, aucune policy UPDATE sur `profiles` → un user **ne peut pas** se passer `plan='pro'` lui-même. Le seul écrivain du plan est `activate_pro` en `service_role`. C'est exactement le bon contrôle.
- **RPCs `SECURITY DEFINER`** avec `set search_path = public` et `revoke from anon` : périmètre correct, pas d'injection de search_path.
- **Clés IA jamais côté client :** tout passe par l'Edge Function. Bon design.
